📖 Engine User Guide & Architecture Overview

Comprehensive documentation on system operations and algorithmic logic.

1. Operation Manual & Workflows

  1. Data Ingestion: Load your datasets (e.g., Beneficiaries, Product Catalogs, IoT Logs) by pasting raw CSV text or utilizing the drag-and-drop file uploader. The system supports multi-source merging natively.
  2. Generic Field Mapping (Critical Step): Map your CSV columns to the engine's core matching parameters.
    • Unique ID: Used for deterministic (exact) matching.
    • Primary Name / Title: The main string identifier (heavily weighted).
    • Dynamic Capture: Any columns not mapped to the core engine are not discarded. They are dynamically swept into the Other Data (Dynamic) payload to ensure zero data loss.
  3. Configure Confidence Thresholds:
    • Auto-Merge Threshold (e.g., 92%): Scores above this line are considered definitive matches. The engine will automatically fuse them into a single record.
    • Review Threshold (e.g., 75%): Scores falling between this value and the auto-merge line are flagged as "Probable Matches" and routed to the Human Adjudication Queue.
  4. Run Resolution Pipeline: Execute the engine. You can monitor the computational steps in real-time via the interactive terminal on the right.
  5. Human-in-the-Loop Adjudication: Navigate to the ⚖ Review Queue tab. Here, human operators act as the final decision-makers to manually Merge or Keep Separate borderline cases, preventing algorithmic false positives.
  6. Export & Interoperability: Export your deduplicated Master Registry to CSV, Excel (XLSX) for business users, JSON for API integrations, or DCMI XML for institutional data archiving.

2. Under the Hood: Python Processing Architecture

Upon pipeline execution, the embedded Python (Pyodide) environment performs an advanced Record Linkage / Identity Resolution sequence entirely on the client-side (Zero-Trust architecture—no data leaves your browser). Here is the lifecycle of a record:

A. Dynamic Schema Ingestion

The engine utilizes Python's csv.DictReader to parse rows. Mapped fields are extracted for the matching algorithm, while all unknown columns are swept into a dynamic dictionary object (*kwargs format), enabling the engine to process highly heterogeneous datasets without strict predefined schemas.

B. Quality & Completeness Scoring

Before matching, each record is evaluated for data quality. The system calculates a completeness ratio by verifying the presence of expected core fields, generating a normalized Quality Score (0-100%).

C. Composite Matching Algorithm

The system compares incoming data against the unified Master Registry using a hybrid approach:

  • Deterministic Match: Immediate 100% score if the Unique IDs match exactly.
  • Fuzzy String Matching: Utilizes difflib.SequenceMatcher (Gestalt Pattern Matching) to measure the Levenshtein-like distance between strings. This accounts for typos, phonetic variations, and missing characters (e.g., "Jon Doe" vs "John Doe").
  • Weighted Aggregation: Variables are scored independently and aggregated based on mathematical weights (e.g., Primary Title similarity contributes heavily, while Secondary Description contributes less) to form the final confidence score.

D. Resolution Matrix & Data Provenance

The final composite score dictates the routing of the data block:

🟢 Auto-Merge
High confidence score. The system automatically performs a merge_into() operation, fusing attributes and updating source tags.
🟡 Human Adjudication
Probable match. Routed to the review_queue. The engine halts automated merging to prevent data corruption.
⚪ Mint New Entity
Low confidence. The engine calls make_master() to instantiate a brand new UUID in the registry.

Audit Trail: Throughout this process, every action is logged into an immutable Audit Log. When fields are overwritten, the engine maintains Data Provenance, logging the timestamp, operator, and origin source of the modification.
